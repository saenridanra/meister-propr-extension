# Implementation Plan: Crawl Configuration Settings

**Branch**: `001-crawl-config-settings` | **Date**: 2026-03-09 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-crawl-config-settings/spec.md`

## Summary

Add a Crawl Configuration section to the existing Meister ProPR project settings hub, allowing administrators to configure automatic PR crawling by entering a reviewer display name. On save, the extension lists the client's crawl configurations, then creates one if none exists for the current project (display name set) or deletes the matching one (display name cleared). The reviewer display name is persisted in extension data storage scoped per project to enable pre-population on load. A new `clientId` field is added to settings, as the backend crawl config API requires it as a URL path parameter. Jest + ts-jest are introduced as the test framework to satisfy the TDD constitution requirement.

## Technical Context

**Language/Version**: TypeScript 5.7 (strict mode)
**Primary Dependencies**: `azure-devops-extension-sdk`, `azure-devops-extension-api`, `axios`, Jest + ts-jest (new, dev-only)
**Storage**: ADO `IExtensionDataService` (Default scope, keyed values)
**Testing**: Jest + ts-jest (new — no prior test setup)
**Target Platform**: Sandboxed browser iframe inside Azure DevOps (project-admin hub)
**Project Type**: Azure DevOps browser extension (single-page, iframe-based)
**Performance Goals**: Settings save completes (including API calls) within the user's reasonable wait tolerance; no explicit latency target beyond standard HTTP round-trip
**Constraints**: No Node.js APIs, no React/Vue, no WebSockets; ADO SDK for all ADO interactions; no secrets in URLs; `clientKey` in headers only
**Scale/Scope**: Single settings page; 2–3 API calls per save; negligible load

## Phase 1: Design

### 1. ExtensionSettings Extension (`src/common/extensionSettings.ts`)

**Changes**:
- Add `clientId: string` to `ExtensionSettings` interface
- Add `clientId` to the `KEYS` constant
- Update `loadSettings()` and `saveSettings()` to include `clientId`
- Add helper `loadProjectCrawlReviewerName(projectId: string): Promise<string>` — loads `crawlReviewerDisplayName_{projectId}`
- Add helper `saveProjectCrawlReviewerName(projectId: string, name: string): Promise<void>` — saves or clears the per-project display name

### 2. Crawl Config API Client (`src/api/crawlConfigClient.ts`)

**Prerequisite**: `src/generated/` must exist (produced by `npm run generate:api`, task T000).

**New file** — thin adapter over the generated `ClientsService`. Types (`CrawlConfigResponse`, `CreateCrawlConfigRequest`, `ApiError`) are imported from `src/generated/`; no type duplication.

```typescript
import { OpenAPI, ClientsService, ApiError } from '../generated';
import type { CrawlConfigResponse, CreateCrawlConfigRequest } from '../generated';

export const DEFAULT_CRAWL_INTERVAL_SECONDS = 300;

function configure(backendUrl: string, clientKey: string): void {
  OpenAPI.BASE = backendUrl.replace(/\/$/, '');
  OpenAPI.HEADERS = { 'X-Client-Key': clientKey };
}

function toError(err: unknown): Error {
  if (err instanceof ApiError) return new Error(err.body?.detail ?? err.message);
  return err instanceof Error ? err : new Error(String(err));
}

export async function listCrawlConfigs(backendUrl, clientKey, clientId): Promise<CrawlConfigResponse[]>
export async function createCrawlConfig(backendUrl, clientKey, clientId, request): Promise<CrawlConfigResponse>
export async function deleteCrawlConfig(backendUrl, clientKey, clientId, configId): Promise<void>
// deleteCrawlConfig treats ApiError(status=404) as success (idempotent)
```

**Auth**: injected via `OpenAPI.HEADERS` before each call — no `X-Ado-Token` required for crawl config endpoints.

### 2a. Crawl Config Orchestrator (`src/settings/crawlConfigOrchestrator.ts`)

**New file** — pure, DOM-free orchestration logic extracted to enable TDD under `testEnvironment: "node"`.

```typescript
export async function runCrawlConfigSave(params: {
  reviewerDisplayName: string; orgUrl: string; projectId: string;
  backendUrl: string; clientKey: string; clientId: string;
}): Promise<void>

export async function loadCrawlConfigState(projectId: string): Promise<string>
```

`runCrawlConfigSave`: calls `listCrawlConfigs` → finds match on `organizationUrl`+`projectId` → creates or deletes → calls `saveProjectCrawlReviewerName`.

`loadCrawlConfigState`: calls `loadProjectCrawlReviewerName(projectId)` and returns the result.

**TypeScript logic additions** (in `src/settings/settings.ts`):
1. On init: acquire `pageContext` → extract `projectId` and `orgUrl`
2. On load: populate `clientId` input from `settings.clientId`; populate `reviewerDisplayName` input by calling `loadCrawlConfigState(projectId)` from `crawlConfigOrchestrator.ts`
3. On save:
   - Save `clientId` alongside `backendUrl` + `clientKey`
   - If `clientId` is missing: skip crawl config operations, show warning
   - Otherwise: call `runCrawlConfigSave(…)` from `crawlConfigOrchestrator.ts`; catch thrown `Error` and display in `statusMsg`
4. Prerequisite guard: if `backendUrl`, `clientKey`, or `clientId` are empty, disable the reviewer display name input and show a "Configure backend settings first" hint
